1. What Is This Project?

GPCSSI-FINAL is a web application that lets you upload server log files (like Apache access logs or Linux system logs) and automatically detects anomalies — unusual, suspicious, or error-related lines — using Artificial Intelligence.

Instead of manually scanning through thousands of log lines, you upload the file to a web page, pick an AI model, and get back a clean report showing only the problematic lines with explanations of why each one is suspicious.

The app was built as part of the GPCSSI program (Google Cybersecurity and Safety Initiative) to demonstrate how Large Language Models (LLMs) can be applied to real-world security monitoring tasks.

Simple Analogy: Imagine you have a 500-page server diary. Reading all of it would take hours. This app is like giving that diary to a smart assistant who quickly flips through every page and flags only the suspicious entries, telling you why each one caught their eye.

2. The Problem It Solves

Why is log analysis hard?

Servers generate thousands of log lines every hour. A typical Apache web server log can have 50,000+ lines in a single day. Manually finding security issues, errors, or unusual patterns in that much text is:

  • Time-consuming: Hours of reading repetitive text
  • Error-prone: Humans miss things after the 1000th line
  • Skill-dependent: You need to know what "normal" looks like for each log format

Why not use traditional tools?

Traditional tools like Splunk or ELK Stack rely on:

  • Rules/Regex: You write patterns like error|failed|denied. But what if an anomaly doesn't match any known pattern?
  • Machine Learning models: These need labeled training data (thousands of "normal" and "anomalous" examples) — which you often don't have.
  • Heavy infrastructure: Setting up Elasticsearch clusters just to scan a log file is overkill.

How this project solves it

Instead of rules or trained models, this app uses Large Language Models (LLMs) — AI models that understand language and context. The LLM reads each log line (or groups of lines) and decides whether it looks anomalous based on its general understanding of server logs. It doesn't need training, doesn't need regex patterns, and works on any text-based log format.

Key Insight: The same LLMs that can write essays and answer questions can also read server logs and tell you if something looks wrong — because they've been trained on vast amounts of text, including technical documentation and code.

Step-by-Step Pipeline

  1. User loads the web app at http://localhost:8000
  2. On page load, JavaScript:
    • Calls GET /api/gpu-status to check NVIDIA GPU + Ollama health
    • Loads saved Google API key from localStorage and auto-validates it
  3. User selects options: Model (Llama/Qwen/Gemini), Mode (Auto/Line-by-Line/Intensive/Chunking), Speed (Balanced/Fast/Max)
  4. User uploads a log file and clicks "Analyze Logs"
  5. Frontend sends POST /upload-log/ with the file + form parameters
  6. Backend decides which detector to use:
    • If model starts with "gemini-" → GeminiDetector
    • Otherwise → OllamaDetector
  7. File is parsed: read, decoded to UTF-8, split into lines (blank lines skipped)
  8. Lines are grouped based on the selected mode (chunk size determined by mode)
  9. LLM processes chunks concurrently (concurrency controlled by speed setting via asyncio.Semaphore)
  10. Results are streamed back via SSE as JSON events: progress updates + final anomaly report
  11. Frontend displays: progress bar, total logs count, anomalies found, and a detailed table

Key Code: Gemini Rate Limiter (src/gemini_detector.py:15-36)

class RateLimiter:
    """Token bucket rate limiter to prevent API burst limits."""
    def __init__(self, requests_per_second=0.5, burst=2):
        self.rate = requests_per_second
        self.burst = burst
        self.tokens = burst

    async def acquire(self):
        async with self._lock:
            now = time.monotonic()
            elapsed = now - self.last_refill
            self.tokens = min(self.burst,
                              self.tokens + elapsed * self.rate)
            if self.tokens >= 1:
                self.tokens -= 1
                return
            wait = (1 - self.tokens) / self.rate
            await asyncio.sleep(wait)

5a. ML Pre-filter Integration (Two-Stage Pipeline)

A pre-filter step has been added to dramatically reduce the number of lines the LLM needs to analyze. Inspired by AICoE/log-anomaly-detector, it uses an unsupervised ML approach before the LLM stage.

Why a pre-filter?

A 50,000-line log file would take the LLM a very long time to process (potentially thousands of API calls). The pre-filter quickly scans all lines and passes only the most "unusual" ones to the LLM, saving time and API costs.

How it works

  1. Feature extraction: Each log line is converted to a 44-dimensional numerical vector (character frequencies, digit ratio, special chars, word stats, etc.)
  2. SOM (Self-Organizing Map): A neural network grid learns the pattern distribution of all log lines. Each node in the grid represents a "typical pattern."
  3. Scoring: Each line gets an anomaly score based on how far it is from its closest SOM node, weighted by how rare that SOM node is.
  4. Thresholding: Lines in the top N% (default: 15%) of scores are flagged as potential anomalies and passed to the LLM.

Two-Stage Pipeline

Raw Log File │ ▼ ┌─────────────────────────────────┐ │ Stage 1: ML Pre-filter (SOM) │ Fast (5s for 56K lines) │ - Feature extraction │ Flags top 10-15% as candidates │ - SOM training + inference │ └──────────┬──────────────────────┘ │ Only flagged lines (~8K / 56K) ▼ ┌─────────────────────────────────┐ │ Stage 2: LLM Analysis │ Slower but targeted │ - Ollama or Gemini │ Explains WHY each line is anomalous │ - Detailed explanations │ └──────────┬──────────────────────┘ ▼ Final Anomaly Report

Integration with log-anomaly-detector

The src/prefilter.py module implements the core algorithm from AICoE/log-anomaly-detector (Word2Vec-style text encoding + Self-Organizing Map) adapted as a lightweight Python class with only numpy as a dependency. The original package is archived and has complex dependencies (gensim, sompy, etc.), so the core logic was reimplemented for seamless integration.

Key Code: SOM Anomaly Scoring (src/prefilter.py)

def _compute_score(self, vector):
    # Distance from line to closest SOM node
    distances = np.sum((self.som - vector) ** 2, axis=2)
    bmu = np.unravel_index(np.argmin(distances), distances.shape)
    min_dist = float(np.min(distances))

    # Penalty for rare BMUs (few lines map here)
    hits = self.bmu_hits[bmu[0], bmu[1]]
    freq_penalty = 1.0 / max(math.sqrt(hits), 0.01)

    # Combined score
    return min_dist * freq_penalty

6. Analysis Modes Explained

The app offers 4 analysis modes that control how log lines are sent to the LLM:

ModeHow It WorksBest ForSpeed
Auto (default) Logs are grouped into chunks of 10 lines each. Each chunk is sent to the LLM at once. The LLM returns a list of anomalies found in that chunk. General purpose — good balance of speed and accuracy Fast (10x fewer API calls)
Line-by-Line Each individual log line is sent separately to the LLM. Highest accuracy but slowest. Small files where accuracy is critical; NOT available for Gemini (too expensive) Slow (1 call per line)
Intensive Larger chunks of 20 lines each. Sends fewer requests per file. Very large files where speed is more important than per-line accuracy Fastest (fewer calls)
Chunking User defines what percentage of total lines each chunk should be. For example, 5% of a 10,000-line file = 500-line chunks. Custom control; fine-tuning for specific file sizes Depends on chunk size

When to use which?

Use Ollama when: Privacy matters, you have no internet, or you want to avoid API costs. Best for learning and experimentation.
Use Gemini when: You need the highest accuracy, faster results, or are analyzing large files. Best for production use.

8. How To Run The Project

Prerequisites

  • Python 3.9+
  • pip (Python package installer)
  • Ollama installed and running (for local models)
  • Google API key (optional, for Gemini models)

Quick Start

# 1. Install Python dependencies
pip install -r requirements.txt

# 2. Set up Ollama (automated script)
# macOS/Linux:
./scripts/setup_ollama.sh
# Windows:
.\scripts\setup_ollama.ps1

# 3. Start the server
uvicorn src.app:app --reload

# 4. Open in browser
open http://localhost:8000

What happens on first load?

The app automatically:

  • Detects your NVIDIA GPU via nvidia-smi
  • Checks if Ollama is running
  • Guides you to configure GPU acceleration if needed
  • Validates any saved Google API key
  • Populates the model dropdown with available options

GPU Configuration

The app forces Ollama to use CUDA (NVIDIA GPU) instead of Vulkan by setting environment variables at startup:

# These are set automatically in src/app.py
os.environ["OLLAMA_VULKAN"] = "0"
os.environ["OLLAMA_LLM_LIBRARY"] = "cuda_v13"

10. Why This Project Stands Out

  1. Zero Infrastructure: Unlike Splunk/ELK that need databases, clusters, and heavy services, this project runs as a single Python process. Install 7 packages and run one command.
  2. No Training, No Rules: Traditional ML-based anomaly detection needs thousands of labeled examples. This project uses pre-trained LLMs — the model already understands what "normal" and "anomalous" look like from its training data.
  3. Privacy-First Design: With Ollama support, you can run the entire application offline with no data leaving your computer. None of the alternative tools offer this in a free, easy-to-use package.
  4. Explains Its Reasoning: The LLM returns natural-language explanations for each anomaly (e.g., "Multiple failed login attempts from same IP in short time"). Traditional tools only show a score or a matched pattern name.
  5. Format-Agnostic: Works on Apache logs, Linux syslogs, error logs, or any text-based log format — without writing format-specific parsers. Traditional tools require grok patterns or regex for each format.

What GPCSSI-FINAL is NOT trying to be

  • It's not a real-time monitoring system (no continuous log tailing)
  • It's not an enterprise SIEM (no user management, no RBAC)
  • It's not a replacement for production monitoring tools

It is a lightweight, educational, developer-friendly tool for quick ad-hoc log analysis using the power of LLMs. It proves a concept: that modern AI can understand server logs without being explicitly trained for it.
